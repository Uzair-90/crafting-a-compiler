### Question No. 17

Translation of a regular expression into an NFA is fast and simple. Cre-
ation of an equivalent DFA is slower and can lead to a much larger
automaton. An interesting alternative is to scan using NFAs, thus obvi-
ating the need to ever build a DFA. The idea is to mimic the operation of
the Close and MakeDeterministic routines (as deﬁned in Section 3.8.2)
while scanning. A set of possible states, rather than a single current state,
is maintained. As characters are read, transitions from each state in the
current set are followed, thereby creating a new set of states. If any state
in the current set is ﬁnal, then the characters read will comprise a valid
token.
Deﬁne a suitable encoding for an NFA (perhaps a generalization of the
transition table used for DFAs) and write a scanner driver that can use
this encoding by following the set-of-states approach outlined previ-
ously. This approach to scanning will surely be slower than the standard
approach, which uses DFAs. Under what circumstances is scanning
using NFAs attractive?

### Solution

### Exercise 17

The exercise asks for an **NFA-based scanner driver** instead of converting the NFA into a DFA. The book says this avoids building a possibly large DFA, but scanning becomes slower because we track a **set of possible NFA states** instead of one DFA state. 

### NFA Encoding

Represent each NFA state like this:

```txt
State:
    transitions: list of (character/class/λ, targetState)
    isFinal: true/false
    tokenType: token name if final
```

Example:

```txt
state 0:
    λ -> state 1
    λ -> state 5

state 1:
    'a' -> state 2

state 2:
    'b' -> state 2
    'c' -> state 3

state 3:
    final = true
    tokenType = TOKEN_X
```

### Scanner Driver

```txt
currentSet = λ-closure({startState})
lexeme = ""
lastFinalSet = empty
lastFinalPosition = -1

while input has characters:
    c = next input character

    nextSet = empty

    for each state in currentSet:
        for each transition from state:
            if transition matches c:
                add targetState to nextSet

    nextSet = λ-closure(nextSet)

    if nextSet is empty:
        break

    consume c
    append c to lexeme
    currentSet = nextSet

    if currentSet contains final state:
        lastFinalSet = currentSet
        lastFinalPosition = current input position
```

After the loop:

```txt
if lastFinalSet is not empty:
    roll input back to lastFinalPosition
    return token from highest-priority final state
else:
    report lexical error
    consume one bad character
```

### Why remember `lastFinalSet`?

Because scanners use the **longest-match rule**.

Example:

```txt
=
==
```

If the scanner sees `=`, that is already a valid token, but it should continue reading to check whether the longer token `==` exists.

So it remembers the most recent final state, but keeps scanning until no transition is possible.

### When is NFA scanning attractive?

NFA scanning is useful when:

```txt
1. Building the DFA would be too expensive.
2. The equivalent DFA may be exponentially larger.
3. The scanner is generated dynamically or only used once.
4. Startup time matters more than scanning speed.
5. Token specifications change frequently.
6. Memory is limited and storing the full DFA is undesirable.
```

Main idea:

```txt
DFA scanner:
    fast scanning
    expensive construction sometimes

NFA scanner:
    simple construction
    smaller representation
    slower scanning
```
