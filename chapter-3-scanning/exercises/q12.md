### Question No. 12

Because Fortran generally ignores blanks, a character sequence contain-
ing n blanks can be scanned as many as 2n diﬀerent ways. Are each
of these alternatives equally probable? If not, how would you alter
the design you proposed in Exercise 11 to examine the most probable
alternatives ﬁrst?

### Solution

### Exercise 12

No, the `2^n` scanning alternatives are **not equally probable**.

Because Fortran ignores blanks, a line with `n` blanks can theoretically be interpreted in many ways, but real programs strongly favor certain interpretations. For example, `DO 10 I = 1 , 10` is much more likely to be a `DO` loop than a strange identifier sequence. The book notes that Fortran scanners may need extended lookahead because blanks are insignificant except inside strings. 

A good scanner design should try the **most likely interpretations first**:

```txt
1. Read the entire statement line into a buffer.
2. Remove insignificant blanks.
3. First try common structured forms:
   - DO statement
   - IF statement
   - assignment
   - CALL statement
4. Use lookahead only where ambiguity remains.
5. If the most likely interpretation fails, backtrack and try the next one.
```

For the `DO` example:

```fortran
DO 10 I = 1 , 10
```

try this first as:

```txt
DO statement
```

because the comma after the initial value strongly suggests a loop.

But:

```fortran
DO 10 I = 1 . 10
```

should be interpreted as:

```txt
assignment to variable DO10I
```

because the period makes `1.10` a real literal.

So the improved Exercise 11 design is:

```txt
Use a ranked/backtracking scanner:
    generate possible tokenizations,
    test the most probable one first,
    fall back only when it fails.
```

In practice, the ranking would be based on Fortran syntax and common statement keywords, not on treating all blank/no-blank choices equally.
