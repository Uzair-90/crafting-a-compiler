### Question No. 16

Consider the following regular expression:
(0 | 1)* 0(0 | 1)(0 | 1)(0 | 1) . . . (0 | 1)
Display the NFA corresponding to this expression. Show that the equiv-
alent DFA is exponentially bigger than the NFA you presented.

### Solution

### Exercise 16

Regular expression: 

```txt
(0 | 1)0(0 | 1)(0 | 1)(0 | 1)...(0 | 1)
```

Assume there are `n` copies of `(0|1)` after the required `0`.

So the language is:

```txt
all binary strings where the (n+1)-th character from the end is 0
```

Example for `n = 3`:

```txt
(0|1)0(0|1)(0|1)(0|1)
```

This accepts binary strings of length `5` whose second character is `0`.

### NFA

The NFA is small because it can “guess” the important `0`.

```txt
        0,1
      +-----+
      |     v
--> q0 ---- q0
      |
      | 0
      v
     q1 --0,1--> q2 --0,1--> q3 --0,1--> ... --0,1--> qn --0,1--> qf*
```

Meaning:

```txt
q0 loops over any prefix.
When it sees a 0, it guesses: "this is the 0 that is n+1 positions from the end."
Then it must read exactly n more binary characters.
```

Number of NFA states:

```txt
n + 2
```

approximately linear.

### Why the DFA is exponentially larger

A DFA cannot guess. It must remember enough of the recent input to know whether the character `n+1` positions back was `0`.

That means the DFA must remember the last `n+1` bits.

Number of possible bit histories:

```txt
2^(n+1)
```

So the equivalent DFA needs exponentially many states.

### Core idea

```txt
NFA:
    can guess which 0 matters
    needs only O(n) states

DFA:
    cannot guess
    must remember all possible recent suffixes
    needs O(2^n) states
```

So this is an example where converting an NFA to a DFA causes **exponential blowup**.
