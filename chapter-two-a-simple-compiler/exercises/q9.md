### Question No. 9

The abstract tree design for ac uses a single node to represent a print
operation ![image](../../images/fig-2.9.png). Consider an alternative design in which the
print operation always has a single id child that represents the variable to
be printed. What are the design and implementation issues associated
with the two approaches?

### Solution

In Figure 2.9, print b is represented as one node:

```C
print
 b
```
That means the print node directly stores the identifier name b.

An alternative design would be:

```bash
print
  |
 id
  |
 b
```
Here, the print node does not store the variable name directly. Instead, it has one child: an id node.

The single-node design is simpler and smaller. It uses fewer AST nodes, less memory, and makes code generation easy because the compiler can directly read the identifier from the print node. However, it treats print differently from other expression-like constructs because the identifier is stored as an attribute instead of as a child node.

The child-node design is more uniform. Since identifiers already have their own AST node type, print can be treated like other operations that have child operands. This can make semantic analysis and code generation cleaner because the compiler can process the child using the same logic it uses for other identifiers. It also makes the design easier to extend later, for example if print is changed to allow expressions like print a + 3. The disadvantage is that it creates an extra node, making the AST slightly larger and requiring a little more traversal logic.

In summary, storing the identifier directly in the print node is compact and efficient, while giving print an id child is more regular, flexible, and easier to extend. For a very small language like ac, the single-node approach is acceptable, but for a growing language, the child-node design is usually better.
