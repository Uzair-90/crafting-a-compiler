### Question No. 13

You are to design the ultimate programming language, “Utopia 2010.”
You have already speciﬁed the language’s tokens using regular expres-
sions and the language’s syntax using a CFG. Now you want to deter-
mine those token pairs that require whitespace to separate them (such as
else a) and those that require extra lookahead during scanning (such as
10.0e-22). Explain how you could use the regular expressions and CFG
to automatically ﬁnd all token pairs that need special handling.

### Solution

This exercise asks for token pairs where the scanner needs:

```txt
1. whitespace separation, or
2. lookahead
```

to distinguish tokens correctly.

---

## (a) Cases requiring whitespace separation

Without whitespace, adjacent tokens may merge into a different valid token.

Examples:

| Intended Tokens | Without Space | Problem                |
| --------------- | ------------- | ---------------------- |
| `int x`         | `intx`        | becomes identifier     |
| `if else`       | `ifelse`      | becomes identifier     |
| `do 10`         | `do10`        | becomes identifier     |
| `123 456`       | `123456`      | becomes one integer    |
| `x y`           | `xy`          | becomes one identifier |

Example:

```txt id="6m5qfk"
float value
```

needs whitespace; otherwise:

```txt id="w4v4v8"
floatvalue
```

is scanned as one identifier.

---

## (b) Cases requiring lookahead

The scanner must inspect future characters before deciding the token.

Examples:

| Input Begins With | Need Lookahead To Distinguish |
| ----------------- | ----------------------------- |
| `<`               | `<` vs `<=`                   |
| `>`               | `>` vs `>=`                   |
| `=`               | `=` vs `==`                   |
| `!`               | `!` vs `!=`                   |
| `/`               | `/` vs `/*` comment           |
| `.`               | `.` vs floating literal       |
| `1.0e`            | exponent continuation         |

Examples:

```txt id="a76v4g"
<
<=
```

Scanner reads `<` then looks ahead one character.

Another:

```txt id="p6ejp2"
/
/*
```

Scanner must look ahead after `/` to determine division operator or comment start.

---

## (c) Multicharacter lookahead examples

Some languages require more than one character:

```txt id="m6hb5t"
>>=
<<=
...
```

or Fortran’s:

```txt id="7j7cyy"
DO 10 I = 1,10
DO 10 I = 1.10
```

where the scanner may need substantial lookahead before deciding tokenization.

---

Main idea:

```txt id="s09w8x"
Whitespace prevents token merging.
Lookahead resolves ambiguous prefixes.
```
