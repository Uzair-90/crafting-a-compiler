### Question No. 7

Deﬁne a token class AlmostReserved to be those identiﬁers that are not
reserved words but that would be if a single character were changed.
Why is it useful to know that an identiﬁer is “almost” a reserved word?
How would you generalize a scanner to recognize AlmostReserved tokens
as well as ordinary reserved words and identiﬁers?

### Solution

Exercise 7 is about recognizing identifiers while excluding reserved words such as `if`, `while`, etc. 

The exercise introduces the token:

```txt
AlmostReserved
```

which matches strings that begin like reserved words but continue with extra characters.

Example:

```txt
if        reserved word
ifx       identifier
while1    identifier
```

A suitable regular expression is:

```txt
(if | while | begin | end | ...)[A-Za-z0-9_]+
```

General form:

```txt
(reserved_word)(letter | digit | _) +
```

Meaning:

* first match a reserved word,
* then require at least one additional identifier character.

Examples accepted:

```txt
ifx
whileLoop
begin1
endif
```

Examples rejected:

```txt
if
while
begin
```

because those are exact reserved words, not identifiers.

Purpose of the exercise:

The scanner can first detect `AlmostReserved` tokens and then classify them as ordinary identifiers instead of reserved words. This avoids ambiguity between keywords and identifiers beginning with keyword prefixes.
