### Question No. 8

When a compiler is ﬁrst designed and implemented, it is wise to con-
centrate on correctness and simplicity of design. After the compiler is
fully implemented and tested, you may need to increase compilation
speed. How would you determine whether the scanner component of a
compiler is a signiﬁcant performance bottleneck? If it is, what might you
do to improve performance (without aﬀecting compiler correctness)?

### Solution

Exercise 8 asks how to check whether the scanner is a performance bottleneck, and how to improve it without changing correctness. 

### Answer

First, do **profiling**:

```txt
Run the compiler on large source programs and measure where time is spent.
```

Use a profiler such as:

```txt
gprof, prof, pixie, qpt
```

If a large percentage of compilation time is spent inside scanner routines, then the scanner is a bottleneck.

To improve scanner performance:

```txt
1. Use a fast scanner generator such as Flex or GLA.

2. Replace character-by-character input with block input.

3. Use double buffering so tokens can cross buffer boundaries safely.

4. Avoid unnecessary copying of characters.

5. Only copy token text when needed, such as for identifiers, numeric literals, and string literals.

6. Process token text directly from the input buffer when possible.

7. Optimize symbol-table lookup for identifiers and reserved words.
```

The main idea:

```txt
Do not guess. Profile first.
Then optimize only the expensive scanner operations.
```

The book specifically says single-character input can be inefficient because each subroutine call may cost many instructions, and block reads plus double buffering can greatly improve scanning speed.
