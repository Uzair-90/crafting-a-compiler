### Question No. 11

Unlike C, C++, and Java, Fortran generally ignores blanks and therefore
may need extensive lookahead to determine how to scan an input line.
A typical example of this is DO 10 I = 1 , 10, which produces seven
tokens, in contrast with DO 10 I = 1 . 10, which produces three to-
kens.
(a) How would you design a scanner to handle the extended lookahead
that Fortran requires?
(b) Lex contains a mechanism for doing lookahead of this sort. How
would you match the identiﬁer (DO10I) in this example?


### Solution

### Exercise 11

Fortran ignores blanks, so the scanner cannot always decide tokens immediately. The book’s example is:

```fortran
DO 10 I = 1 , 10
```

This is a `DO` loop, producing tokens like:

```txt
DO  INT(10)  ID(I)  =  INT(1)  ,  INT(10)
```

But:

```fortran
DO 10 I = 1 . 10
```

is treated like an assignment to identifier `DO10I`, producing roughly:

```txt
ID(DO10I)  =  REAL(1.10)
```

because blanks are ignored. 

### (a) Scanner design

Use **extended lookahead with buffering**:

```txt
1. Read the whole statement line into a buffer.
2. Remove blanks outside strings.
3. Try to recognize special Fortran forms such as DO statements.
4. Look ahead until the scanner sees the key disambiguating character:
   comma "," means DO loop
   period "." means assignment
5. Return tokens from the selected interpretation.
```

So for:

```fortran
DO 10 I = 1 , 10
```

after removing blanks:

```txt
DO10I=1,10
```

the scanner sees the comma after `1`, so it treats `DO` as a keyword.

For:

```fortran
DO 10 I = 1 . 10
```

after removing blanks:

```txt
DO10I=1.10
```

the scanner sees the period, so it treats `DO10I` as one identifier.

### (b) Lex lookahead rule

Lex has trailing context using `/`.

To match the identifier `DO10I` only when it is followed by assignment context, use a rule like:

```lex
DO[0-9]+[A-Za-z][A-Za-z0-9]* / [ \t]*=
{
    return ID;
}
```

For this example, it matches:

```txt
DO10I
```

when followed by:

```txt
=
```

But to distinguish the loop from assignment, a more Fortran-specific lookahead rule would check what comes after the `=`:

```lex
DO[ \t]*[0-9]+[ \t]*[A-Za-z][A-Za-z0-9]*[ \t]*=[ \t]*[0-9]+[ \t]*\.
```

This identifies the assignment-style case:

```fortran
DO 10 I = 1 . 10
```

and lets the scanner return:

```txt
ID(DO10I)
```

Core idea: **buffer the line, ignore blanks, and use lookahead until comma/period tells whether `DO` is a keyword or part of an identifier.**
