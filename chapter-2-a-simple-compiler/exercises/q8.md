### Solution

#### Extending the `ac` Language to Allow Interspersed Declarations and Statements

The original `ac` grammar requires all declarations to appear before all executable statements:

```text
Prog → Dcls Stmts $
```

This exercise extends the language so that declarations and executable statements may appear in any order. However, an identifier cannot be used in an executable statement until it has been declared.

---

#### (a) Modified CFG

To support interspersed declarations and statements, the grammar can be rewritten so that a program consists of a sequence of general program items.

#### Modified Grammar

```text
Prog  → Items $

Items → Item Items
      | λ

Item  → Dcl
      | Stmt

Dcl   → floatdcl id
      | intdcl id

Stmt  → id assign Val Expr
      | print id

Expr  → plus Val Expr
      | minus Val Expr
      | λ

Val   → id
      | inum
      | fnum
```

---

#### Example Valid Program

```text
intdcl x
x assign 5
floatdcl y
print x
$
```

This program is valid because:
- declarations and statements are intermixed
- identifiers are declared before use

---

#### (b) Revisions to the AST Design

In the original `ac` AST design:
- declarations and statements were stored separately
- because all declarations appeared before executable statements

The original structure was conceptually similar to:

```text
Program
 ├── Declarations
 └── Statements
```

This design no longer works well because declarations and statements can now appear in arbitrary order.

---

#### Revised AST Design

The AST should instead preserve the exact order of program items:

```text
Program
 └── ItemList
      ├── DclNode
      ├── AssignNode
      ├── DclNode
      └── PrintNode
```

In this revised design:
- declarations become ordinary AST nodes
- declarations and executable statements coexist in the same ordered list
- program order is explicitly preserved

---

#### (c) Effect on Semantic Analysis

The semantic analysis phase becomes more order-sensitive.

---

#### Original Semantic Analysis

Previously:
1. all declarations appeared first
2. compiler built the symbol table completely
3. executable statements were analyzed afterward

This separation was simple because every variable was guaranteed to be declared before any statement executed.

---

#### New Semantic Analysis Requirements

Now the compiler must process the AST sequentially from top to bottom.

#### When a declaration is encountered:
- insert the identifier into the symbol table

#### When a statement is encountered:
- verify that every identifier used has already been declared

---

#### Example Semantic Error

```text
print x
intdcl x
$
```

This program must produce a semantic error because:
- `x` is used before it is declared

---

#### Main Consequence

Symbol-table construction and semantic checking can no longer be completely separated.

Instead:
- declarations and statements must be analyzed in program order
- semantic analysis becomes incrementally stateful
- the symbol table evolves as the compiler traverses the AST
