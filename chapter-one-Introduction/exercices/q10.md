### **Question No. 10**

Assume you have a source program P. It is possible to transform P
into an equivalent program P by reformatting P (by adding or deleting
spaces, tabs, and line breaks), systematically renaming its variables (for
example, changing all occurrences of sum to total), and reordering the
deﬁnition of variables and subroutines.
Although P and P are equivalent, they may well look very diﬀerent.
How could a compiler be modiﬁed to compare two programs and de-
termine if they are equivalent (or very similar)? In what circumstances
would such a tool be useful?


### **Solution**

A compiler could be modified to compare two programs by first converting them into a normalized intermediate representation such as an abstract syntax tree or control-flow graph, applying systematic renaming of variables (alpha-conversion), and performing semantic-preserving transformations such as constant folding, dead code elimination, and loop normalization. The resulting structures can then be compared using techniques such as graph isomorphism, symbolic execution, or logical equivalence checking using formal methods. Such a tool would be useful for verifying compiler optimizations, detecting code plagiarism, validating program refactoring, and ensuring software correctness in safety-critical systems. However, exact program equivalence is undecidable in general, so practical systems rely on approximate or restricted equivalence checking rather than full formal proof of equivalence.
